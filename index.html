How Copilot Studio Connects to 21V Finance and Operations MCP

This guide explains the working integration pattern used to connect Microsoft Copilot Studio to a 21Vianet Dynamics 365 Finance and Operations MCP endpoint. It covers the hosted MCP proxy pattern, the key Copilot Studio configuration values, the Azure deployment shape, and the token refresh model required to keep the connection alive.

Why a Proxy Is Needed

The 21V Finance and Operations MCP endpoint runs behind China sovereign cloud identity infrastructure. In practice, direct client connectivity is not reliable enough for Copilot Studio because the runtime must successfully initialize MCP, retrieve tools, and keep backend authorization valid over time.

The working solution is a hosted MCP proxy. Copilot Studio connects to the proxy over HTTPS, and the proxy connects to the real 21V F&O MCP backend with a valid bearer token. The proxy is also responsible for refreshing that backend token automatically.

Token Refresh Model

The hosted proxy is now designed to refresh the backend 21V token automatically. This avoids the earlier behavior where the proxy worked for a short time and then failed when the access token expired.

  1. The proxy starts with the current access token and a valid refresh token.
  2. Before the cached token expires, the proxy calls the 21V token endpoint.
  3. A new access token is issued and persisted to the site-local cache file.
  4. The proxy continues forwarding MCP traffic without requiring manual updates in App Service.

What Makes the Integration Stable

  • The public endpoint remains constant for Copilot Studio.
  • The proxy stores backend MCP session identifiers and reuses them when needed.
  • The proxy refreshes the backend bearer token instead of relying on a manually pasted token.
  • Copilot Studio only needs a normal MCP server URL and does not need to understand 21V identity details.

Implementation Flow

  1. Deploy the MCP proxy to Azure App Service.
  2. Configure the backend URL, tenant ID, client ID, token endpoint, scope, current access token, and refresh token.
  3. Verify the proxy health endpoint returns status: ok.
  4. Verify MCP initialize and tools/list against the public MCP URL.
  5. Add the MCP server in Copilot Studio with the public MCP URL.
  6. Confirm tool discovery succeeds.
  7. Run a narrow business query such as metadata lookup or purchase-order discovery.

Validation Checks

  • GET /health returns 200
  • POST /mcp initialize returns 200
  • POST /mcp tools/list returns the 21V F&O tool set
  • Copilot Studio shows the discovered tools under the MCP tool page

Security Notes

This public guide intentionally includes only non-secret configuration values. Do not publish access tokens, refresh tokens, API keys, or any tenant-specific secret material in a public repository. If this solution is promoted beyond testing, inbound authentication and stronger operational controls should be added to the hosted proxy.
