How Copilot Studio Connects to 21V Finance and Operations MCP

This guide explains the production-style integration pattern used to connect Microsoft Copilot Studio to a 21Vianet Dynamics 365 Finance and Operations MCP endpoint. It shows the hosted MCP proxy design, the exact Copilot Studio values, the Azure configuration values, and the automatic token refresh model that keeps the connection stable.

Architecture at a Glance

The design deliberately separates Copilot Studio from sovereign-cloud token handling. Copilot Studio only needs a public MCP URL. The hosted proxy owns the China-cloud backend token lifecycle and MCP session continuity.

Copilot Studio stays simple

It only needs one public MCP URL and does not need sovereign-cloud identity logic.

Azure proxy absorbs complexity

Session replay, backend forwarding, and public endpoint stability live in the proxy layer.

Token refresh is isolated

The 21V token lifecycle remains private and is renewed without manual access-token replacement.

ERP tools stay native

The final tool execution still happens on the real 21V F&O MCP backend.

Why a Proxy Is Required

Sovereign Cloud Identity Boundary

The 21V endpoint is protected by the China sovereign identity stack. A normal Copilot Studio endpoint integration should not be expected to own those token exchange details directly.

Stable Public MCP Surface

The proxy gives Copilot Studio one stable MCP URL while hiding backend token issuance, token refresh, and session handling complexity.

Automatic Renewal

Early testing worked for a short time and then failed because a static access token expired. The proxy now supports refresh-token based renewal to prevent that operational failure mode.

Protocol Compatibility

The proxy also maintains backend MCP session state so the 21V server can continue serving tools even when the client does not replay the backend session header exactly as expected.

Automatic Token Refresh Flow

01

Load Current Token State

The proxy reads the current cached access token and expiry time from the persisted token cache path.

02

Check Expiry Window

If the access token is near expiry, the proxy does not wait for an MCP failure. It refreshes proactively.

03

Exchange Refresh Token

The proxy posts to the 21V token endpoint and requests a new access token using the configured refresh token.

04

Persist and Continue

The refreshed access token and latest refresh token are written back to the cache file so App Service restarts can continue using the same lifecycle.

Why the New Model Is More Stable

  • The public MCP URL remains unchanged for Copilot Studio.
  • The proxy owns backend sovereign-cloud token renewal.
  • The proxy stores backend MCP session identifiers and can reuse them for later MCP calls.
  • Operationally, the design removes the need to manually paste a fresh access token into Azure App Service every hour.
  • Copilot Studio only needs a standard MCP server configuration and does not need to know any 21V-specific identity details.

Implementation Flow

A

Deploy the Hosted Proxy

Deploy the .NET MCP proxy to Azure App Service and expose a stable HTTPS URL.

B

Configure 21V Backend Values

Set backend URL, tenant ID, client ID, token endpoint, scope, current access token, and refresh token in App Service settings.

C

Validate the Proxy

Verify GET /health, POST /mcp initialize, and POST /mcp tools/list all succeed.

D

Add the MCP Server in Copilot Studio

Use the public MCP URL and the published values from this guide, then confirm tool discovery succeeds.

Validation Checklist

  • GET /health returns 200
  • POST /mcp initialize returns 200
  • POST /mcp tools/list returns the F&O tool catalog
  • Copilot Studio shows the discovered MCP tools under the tool definition page
  • The proxy continues to work after the original bootstrap access token would normally have expired
